Git 入门 15分钟速成

建立版本库

Git 能够让不同的人在同一时间在同一份文档 (通常是程序代码) 上协同工作，而不需要等别人把事情做完自己再继续。此就是“版本控制系统”的作用。

下面让我们来开始练习 Git 的使用吧。

先安装 Git，然后配置下个人的用户名和邮箱。

$ sudo apt-get install git    # Ubuntu

$ git config --global user.name "Your Name"    # 姓和名拼音首字母大写，中间空一格
$ git config --global user.email "yourname@tp-link.net"

git config 后面带 --global 参数表示设置对本 Unix 账号全局有效。

然后我们先建立一个 “远程版本库”，因为是练习，所以远程库就建立在本地，模拟一下远程库。在实际的开发中，我们的远程库是放在 Gerrit 服务器上的，作为中心版本库使用，所有的开发者都围绕中心版本库工作，通过 SSH 协议和远程库交互。

输入如下命令在用户目录里新建目录并将其初始化一个 Git 裸版本库:

$ mkdir ~/try_git.git
$ cd ~/try_git.git
$ git init --bare

Initialized empty Git repository in /home/yourname/try_git.git/

现在我们已经拥有了一个空的裸版本库。 --bare 选项告诉 Git，需要建立的是一个裸版本库，所谓裸库的意思是里面只放版本库相关的数据，不存放代码。中心版本库必须是裸库。如果你仅仅是想本地建库，把本地一些文件纳入版本控制供自己来管理，那么初始化的时候不必加上 --bare 选项。这里之所以要加，是因为稍后要模拟两个工程师和这个中心库交互。

下面我们模拟一位开发者，在名为 "octobox" 的目录中进行操作。请先克隆刚才建立的远程库为一个本地库，然后进入此目录。

$ cd ~
$ git clone ~/try_git.git octobox

Cloning into 'octobox'...
warning: You appear to have cloned an empty repository.
done.

所谓 “克隆” 的意思是将远程库的所有数据完整地复制。克隆时不带 --bare 参数，则克隆成的库是常规库，不是裸库，克隆完毕后会检出代码。

下面切换到此目录中：

$ cd octobox

正如 Git 在命令行输出中所告诉我们的那样，我们的 "octobox" 目录在 .git 目录下已经有了一个空的版本库，这是一个隐藏的目录，里面有一些 Git 生成的文件。整个目录结构是这个样子的：

.
└── .git
    ├── branches
    ├── config
    ├── description
    ├── HEAD
    ├── hooks
    ├── info
    │   └── exclude
    ├── objects
    │   ├── info
    │   └── pack
    └── refs
        ├── heads
        └── tags

.git 目录以外的部分是工作区的数据，.git 目录内部存放的是版本库和暂存区的数据。

小知识

工作区: 存放代码的地方。暂存区：存放准备提交到版本库的数据版本库：存放版本控制数据的地方，相当于一个数据库。

检查状态

下面，让我们输入 git status 命令来看下我们的项目所处的状态：

$ git status

On branch master

Initial commit

nothing to commit (create/copy files and use "git add" to track)

小知识

.git 目录是一个隐藏目录，在输入某些 Git 命令后，此目录的内容会发生变化，你可以结合 《Pro Git》中有关 Git 对象的介绍，观察其中的变化，加深对 Git 的理解。

新增文件准备提交

下面在 octobox 目录新建一个名为 octocat.txt 的文件。

$ echo "A Tale of Two Octocats" > octocat.txt

此时的目录结构是：

.
├── .git
└── octocat.txt

再次执行 git status 命令看下项目的状态发生了什么变化：

$ git status

On branch master

Initial commit

Untracked files:
  (use "git add <file>..." to include in what will be committed)

    octocat.txt

nothing added to commit but untracked files present (use "git add" to track)

小知识

提示: 好的习惯是经常执行 git status。毕竟有的时候状态已经发生了变化而你并没有察觉。

将变更添加到暂存区

好，看起来我们的 Git 版本库工作正常。请注意为什么 Git 把 octocat.txt 标记为 "untracked (未跟踪)"？ 这意味着 Git 认为 octocat.txt 是一个新文件，还没有将其纳入版本库控制。

要让 Git 开始跟踪 octocat.txt 的变化，首先我们需要使用 git add 命令将其加入暂存区。

$ git add octocat.txt

小知识

状态分下面几种：

已暂存: 文件已经做好了被提交的准备。

未暂存: 改动的文件还没有做好被提交的准备。

未跟踪: 文件还没有被 Git 跟踪。 这通常意味着这个文件是新建的文件。

已删除: 已被跟踪的文件已删除，正在等待从版本库中删除。

检查变更

好，现在 Git 已经开始跟踪 octocat.txt 文件，让我们再次执行 git status 命令来看看现在处于什么状态：

$ git status

On branch master

Initial commit

Changes to be committed:
  (use "git rm --cached <file>..." to unstage)

    new file:   octocat.txt

小知识

add all: 你可以使用 git add -A . 命令，点(.)号代表当前目录，也包括子目录下的文件。而 -A 选项使得即便是文件新增/删除，新增/删除的动作也会被加入暂存区，准备提交。

git reset: 你可以使用 git reset -- <filename> 来取消被加入暂存区中的文件和目录的暂存操作。这条命令做的事情就是用版本库中的文件来恢复暂存区中的文件，而因为版本库中并没有这个文件，所以这个覆盖的动作可以达到 “取消暂存” 的目的。

提交

注意到为什么 Git 将文件的状态标记为 Changes to be committed？这里列出的文件是在暂存区中，还没有存入版本库中。当我们将其存入版本库之前，我们可以新增或修改暂存区中的这些文件。

要将已暂存的变更存入版本库，我们需要执行下面的命令将其提交到本地版本库，并在 -m 选项后面指定的提交主题中简述做出的改动。

(在实际开发中，如果这个提交是需要发布到中心版本库的，那么提交说明需要写详细些，这时我们通常不用 -m 选项，使得 Git 进入一个提交说明的输入界面，在这个界面中输入详细的提交信息。)

下面让我们执行下面的命令：

$ git commit -m "Add cute octocat story"

[master (root-commit) 51115c9] Add cute octocat story
 1 file changed, 1 insertion(+)
 create mode 100644 octocat.txt

小知识

暂存区: 我们准备提交到 Git 版本库的改动放在这里。

提交： 一个 "提交" 是版本库的一个快照。如果我们需要回头查看之前自己或者他人做出的的变更，我们可以看到一个个提交组成了历史变更的漂亮的时间线。

添加所有变更

很好！下面你可以使用通配符来将同类型的多个文件加入暂存区。 下面先在工作目录下创建几个文本文件，将一些文件放在新建的 "octofamily" 的目录下，在 "octobox" 的根目录下也放一些。

$ echo "A Blue Octocat" > blue_octocat.txt
$ echo "A Red Octocat" > red_octocat.txt
$ mkdir octofamily
$ echo "A baby Octocat" > octofamily/baby_octocat.txt
$ echo "A Momma Octocat" > octofamily/momma_octocat.txt

目录结构变为：

.
├── .git
├── blue_octocat.txt
├── octocat.txt
├── octofamily
│   ├── baby_octocat.txt
│   └── momma_octocat.txt
└── red_octocat.txt

幸运的是，我们可以使用带通配符参数的 git add 命令将它们一次性加到暂存区中。别忘了输入引号 (') !

$ git add '*.txt'

小知识

通配符: 当使用通配符的时候，我们需要加上引号，单引号或双引号均可。以便 Git 能够在 Shell 之前处理通配符参数。如果没有引号，我们的 Shell 会在当前目录下搜索符合通配符的文件而不包含子目录，这样 Git 就会接受 Shell 找到的文件列表，那么 octofamily 目录下的文件就不会被加到暂存区了。

提交所有变更

好，你已经将所有的文本文件添加到暂存区了，你可以执行 git status 命令来查看将要提交的文件列表。

如果看起来正常，就继续执行：

$ git commit -m 'Add all the octocat txt files'

[master 9a76150] Add all the octocat txt files
 4 files changed, 4 insertions(+)
 create mode 100644 blue_octocat.txt
 create mode 100644 octofamily/baby_octocat.txt
 create mode 100644 octofamily/momma_octocat.txt
 create mode 100644 red_octocat.txt

小知识

检查所有的东西！

当使用通配符的时，在准备提交的时候，你必须非常小心。请确保在准备提交之前用 git status 命令检查暂存区中暂存了哪些文件，这种做法可以让你确保你要提交的东西是符合自己的预期的。

查看历史

目前为止我们已经做了一些提交。 现在让我们浏览一下我们做出了哪些改动。

好在有了 git log，能够将 Git 提交的历史输出。提交历史记录了我们到目前为止已经提交的变更。为了了解我们之前做了什么提交，执行：

$ git log

commit 9a7615069a9b8521faa5fcb1404f91c007c10c91
Author: Your Name <yourname@tp-link.net>
Date:   Tue Oct 6 14:44:19 2015 +0800

    Add all the octocat txt files

commit 51115c92c5961480e46c4d472fd23e6a9d73c55c
Author: Your Name <yourname@tp-link.net>
Date:   Tue Oct 6 14:40:28 2015 +0800

    Add cute octocat story

commit 后面的16进制 SHA-1 哈希值是为 “提交 ID”，SVN 的提交 ID 是一个数值，而 Git 是哈希值，这是为了保证提交 ID 的唯一性。

小知识

更有用的 log: 使用 git log --summary 可查看关于每个提交的更多的信息。你可以从中看到哪个提交新增了哪些文件，删除了那些文件。通过这个命令可以很好地了解关于项目演进历史的概览。

远程库的位置

执行如下命令查看这个本地库和什么远程库关联。

$ git remote -v

origin  /home/yourname/try_git.git (fetch)
origin  /home/yourname/try_git.git (push)

这说明，本地版本库用于和注册名为 origin 的远程版本库交互。origin 远程库的 URL 是 /home/yourname/try_git.git ，接下来拉取提交的源头和推送提交的目的地，就是到这个 URL。

小知识

git remote: 远程库典型的名字是 origin。你可以将其重命名，Git 不会关心你把远程版本命名为什么名字。主版本库放在远程会更安全，本地机器有可能遇到各种意外而使得本地版本库损坏。所以后面在使用 Gerrit 的时候，建议时不时将本地提交推送到 Gerrit 服务器上挂着，暂不评审，这样可以达到备份之目的。

推送到远程

Push 命令告诉 Git ，当一切准备就绪之后，我们的提交将放在哪里。让我们将我们的本地变更推送到我们的 origin 远程库 (实际在本地硬盘上模拟)。

我们的远程库的名字叫 origin，而默认的本地分支名是 master。下面继续，将它们推送上去!

$ git push origin master

Counting objects: 10, done.
Delta compression using up to 4 threads.
Compressing objects: 100% (4/4), done.
Writing objects: 100% (10/10), 741 bytes | 0 bytes/s, done.
Total 10 (delta 0), reused 0 (delta 0)
To /home/yourname/try_git.git
 * [new branch]      master -> master
Branch master set up to track remote branch master from origin.

从远程拉取

假设其他人在我们的项目上工作，他们拉取了你的变更，做了他们自己的提交，然后推送到远程库中。下面我们新开一个目录模拟他人的操作。

$ cd ~
$ git clone ~/try_git.git
$ cd try_git
$ git config user.name "His Name"
$ git config user.email "hisname@tp-link.net"
$ echo "A yellow Octocat" > yellow_octocat.txt
$ git add yellow_octocat.txt
$ git commit -m 'Add the yellow octocat txt file'
$ git push origin master：master

上述命令中，clone 命令后面不设置本地目录名，那么就会克隆到和远程库名同名的目录下 (目录名去掉后面的 .git)。git config 后面不带 --global 选项表示此设置仅对当前版本库有效，覆盖全局的设置。

下面切换回原来的本地库：

$ cd ~/octobox

我们可以检查远程版本库上的这些更新，然后通过以下命令将新的更新拉取下来：

$ git pull --rebase origin master

小知识

git stash: 有些时候，当你将要从远程拉取时，你本地还有一些没提交的东西。除了提交之外，还有一个选择是储藏 (Stash) 这些变更。

使用 git stash 命令来储藏你的变更， 当 pull 完毕时，使用 git stash apply来重新应用这些变更。

差异比较

下面，我们修改下 octocat 目录里的东西，新增一个文件，再修改一个文件。

$ echo "A Tale of Two Octocats and an Octodog" > octocat.txt
$ echo "woof" > octofamily/octodog.txt

目录结构变为：

.
├── .git
├── blue_octocat.txt
├── octocat.txt
├── octofamily
│   ├── baby_octocat.txt
│   ├── momma_octocat.txt
│   └── octodog.txt
└── red_octocat.txt

现在让我们用 git diff 命令看一下工作区和我们最近一次提交的差异是什么。

在这个例子里，我们想要对比的是我们最近的一次提交，我们可以用 HEAD 头指针来引用最新的提交。

$ git diff HEAD

diff --git a/octocat.txt b/octocat.txt
index 7d8d808..e725ef6 100644
--- a/octocat.txt
+++ b/octocat.txt
@@ -1 +1 @@
-A Tale of Two Octocats
+A Tale of Two Octocats and an Octodog

小知识

HEAD： HEAD 称为头指针，这是一个指针。HEAD 在默认情况下指向你最近的提交，在执行一些 Git 命令后，HEAD 的指向会发生改变，但一定是指向 git log -n1 命令列出的提交。我们能够用 HEAD 来快速引用提交，而不需要查看提交的 ID 值。

已暂存的差异

diff 的另一大用法是查看已经被加入暂存区的文件的变化。记住，暂存的文件就是我们想要告诉 Git 这些是准备被提交的东西。

让我们用 git add 命令来暂存 octofamily/octodog.txt 文件，此文件是我刚刚为你增加的。

$ git add octofamily/octodog.txt

小知识

提交礼仪: 请将每次提交限定于完成一次逻辑功能。并且可能的话，适当地分解为多次小更新，以便每次小型提交都更易于理解。

使用 git diff 可以很好地预览你已经做出的变更，并让你小心地往暂存区里添加文件，分开提交。

好，下面继续，执行添加了 --cached 参数的 git diff 命令，来查看已经暂存的变更。你应该能够看到 octodog.txt 已被创建。

$ git diff --cached

diff --git a/octofamily/octodog.txt b/octofamily/octodog.txt
new file mode 100644
index 0000000..cfbc74a
--- /dev/null
+++ b/octofamily/octodog.txt
@@ -0,0 +1 @@
+woof

重置暂存区

现在 octodog 已经成为家族的一员，octocat 感到很失落。因为我们喜爱 octocat 胜于 octodog，所以我们把 octodog.txt 移除，让 octocat 喜笑颜开。

你可以使用 git reset 命令将版本库的内容恢复到暂存区，从而达到将文件取消暂存的目的。继续，将 octofamily/octodog.txt 移除。

$ git reset octofamily/octodog.txt

Unstaged changes after reset:
M   octocat.txt

git reset 成功地将 octodog.txt 取消暂存，你会注意到，它还在工作区，并不会从本地删除。

撤销

使用 git checkout -- <target> 命令可以将文件变成暂存区中此文件的内容。因为 octocat.txt 自上次提交以来都没有加入过暂存区，所以暂存区的 octocat.txt 和版本库 HEAD 的 octocat.txt 一致。下面继续，把我们对 octocat.txt 所做的修改去掉，回到修改前的样子。

$ git checkout -- octocat.txt

小知识

-- : 也许你会感到奇怪，为什么我要在这个地方使用 --? 如果省略了它，Git 一样可以工作得很好呀。需要注意的是，如果你有一个分支的名字恰好叫做 octocat.txt，在文件参数前面加上 -- 可以保证 Git 依然可以撤销文件，而不是切换到这个同名的分支上。

开分支

当开发者正在开发某个 feature 或者修复某个 bug 时，他们经常会为代码创建一个拷贝 （也就是我们所说的分支），然后在上面做一些提交。当工作完成之后，将这个分支合并回 master 分支。

我们想移除这些令人讨厌的 octocat，所以我们创建了一个名为 clean_up 的分支，在这个分支上来做这件事情：

$ git branch clean_up

小知识

开分支是当你想要同时在多个 feature 上开发时自然而然要做的事情。你总不想在 master 上一路走到底，以至于让 Feature A 开发到一半 的时候切换到 Feature B， 也开发到一半又切换来切换去吧。

更好的做法是分隔两个代码基线到两个“快照” (分支) 上。然后分别在两个分支上工作。一旦工作完成，就把分支合回主干，然后将其推送到远程服务器。主干仅仅是用来推送代码，真正的开发是在各个本地分支上。

切换分支

很好！现在，当你输入 git branch 命令时，你会看到两个本地分支：一个名为 master 的主分支，而你新建的分支名为 clean_up。

你可以使用 git checkout <branch> 命令来切换分支。现在试试切换到 clean_up 分支：

$ git checkout clean_up

Switched to branch 'clean_up'

小知识

如何一次性搞定

你可以使用:

$ git checkout -b new_branch

这条命令会先创建新分支然后马上切换到新分支上，等效于相继执行下面两条命令：

$ git branch new_branch && git checkout new_branch

移除所有的东西

好了，下面你正处在 clean_up 分支，你可以用 git rm 命令最终移除这些令人讨厌的 octocats。这个命令不仅仅是从本地删除这些文件，也会将这个删除的动作加入暂存区。

再次使用通配符，将 octocats 一网打尽。 输入：

$ git rm '*.txt'

rm 'blue_octocat.txt'
rm 'octocat.txt'
rm 'octofamily/baby_octocat.txt'
rm 'octofamily/momma_octocat.txt'
rm 'red_octocat.txt'

小知识

移除所有的东西!

移除一个文件简单，但要是想移除整个目录呢？你可以在 git rm 命令中使用 -r 选项来实现:

$ git rm -r folder_of_cats

这条命令将递归地移除指定的目录以及下面所有的目录和文件。

提交分支上的变更

现在你已经移除了所有的 cats，你需要提交这些变更。

你可以执行 git status 命令来检查你将要提交的东西。

$ git commit -m "Remove all the cats"

[clean_up 223bd1f] Remove all the cats
 6 files changed, 6 deletions(-)
 delete mode 100644 blue_octocat.txt
 delete mode 100644 octocat.txt
 delete mode 100644 octofamily/baby_octocat.txt
 delete mode 100644 octofamily/momma_octocat.txt
 delete mode 100644 red_octocat.txt
 delete mode 100644 yellow_octocat.txt

小知识

'-a' 选项

如果你碰巧删除了一个文件，而没有使用 git rm， 你可以发现，你依然需要使用 git rm 来从工作区中删除文件。你也可以使用在 git commit 时使用 -a 选项，来保存这个本地删除的步骤。这会在提交的时候自动删除文件。git commit -am "Delete stuff"

切换回 master 分支

很好，你已经完成了对 cat 的清理，修复了这个 bug。你现在需要切换回 master 分支，你可以衍合或合并你在 clean_up 分支上的修改，这里使用的方式是 衍合 (rebase) 而非 合并 (merge)命令，是为了保证提交的线性化。

继续，切换回 master 分支：

$ git checkout master

Switched to branch 'master'
Your branch is up-to-date with 'origin/master'.

准备衍合

很好，机会来了，你需要衍合你的修改，从 clean_up 分支衍合回 master 分支。深呼吸一下，没那么吓人。

先用下面的命令查看 log，在 log 中标记了 HEAD 和分支指向的位置：

$ git log --oneline --all --graph --decorate

* 223bd1f (clean_up) Remove all the cats
* 2878653 (HEAD -> master, origin/master) Add the yellow octocat txt file
* 9a76150 Add all the octocat txt files
* 51115c9 Add cute octocat story

当我们在 master 分支上准备就绪，我们告诉 Git，我们要将 clean_up 分支衍合到这里：

$ git rebase clean_up

git rebase clean_up
First, rewinding head to replay your work on top of it...
Fast-forwarded master to clean_up.

再次查看 log：

git log --oneline --all --graph --decorate

* 223bd1f (HEAD -> master, clean_up) Remove all the cats
* 2878653 (origin/master) Add the yellow octocat txt file
* 9a76150 Add all the octocat txt files
* 51115c9 Add cute octocat story

小知识

合并冲突

衍合和合并冲突会发生在同时修改同一个文件时。很多人害怕发生冲突。别怕，没必要对此感到恐惧。你需要做的事情仅仅是决定选用哪段代码。

冲突的处理超过了本课程的范围，如果你对此感兴趣，请阅读 《Pro Git》的相关章节，看看如何来处理冲突。

保持事情干净

恭喜! 你刚刚完成了首个 bug 修复并衍合。剩下需要做的事情是，清理一下，因为你已经在 clean_up 分支上工作完毕，所以就不再需要它了。

你可以使用 git branch -d <branch name> 命令来删除分支。继续，删除 clean_up 分支:

$ git branch -d clean_up

Deleted branch clean_up (was 223bd1f).

小知识

强制删除

如果你已经在一个 feature 分支上工作，然后你决定不要这个 feature 了。你决定删除这个分支，因为你决定砍掉这个 feature，不再开发。你会发现 git branch -d bad_feature 执行失败。这是因为 -d 选项不允许你删除没有合并的分支。你可以在命令中加上 --force (-f) 选项或者使用结合了 -d 和 -f 选项的 -D 来强制删除分支。

最后的推送

现在，最后一个步骤是，为目前已经完成的事情自豪！你在 Git 的学习方面已经收获颇丰！你剩下需要做的事情是将所有完成的东西推送到远程库上，推送完成后就好啦！

$ git push origin master:master

Counting objects: 2, done.
Delta compression using up to 4 threads.
Compressing objects: 100% (1/1), done.
Writing objects: 100% (2/2), 202 bytes | 0 bytes/s, done.
Total 2 (delta 0), reused 0 (delta 0)
To /home/yourname/try_git.git
   2878653..223bd1f  master -> master

master:master 中 : 前面的 master 是本地的源分支，后面的 master 是远程的目的分支。这条命令表示用本地的 master 分支来更新到远程的 master 分支，远程的 master 分支之前指向的提交是 2878653，而本地的 master 分支是指向 223bd1f，那么更新之后 2878653 到 223bd1f 之间新增的提交就被推送到远程啦!

如果远程库是在 Gerrit 上托管，那么如果是要想把提交推送到 Gerrit 上评审而不直接入库的话， master:master 就需要修改为 master:refs/for/master。

查看下 log:

$ git log --oneline --all --graph --decorate
* 223bd1f (HEAD -> master, origin/master) Remove all the cats
* 2878653 Add the yellow octocat txt file
* 9a76150 Add all the octocat txt files
* 51115c9 Add cute octocat story
